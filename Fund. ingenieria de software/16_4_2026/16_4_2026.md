### estimación

	estimar en el desarrollo de software es una de as tareas mas complejas y criticas y se las puede tomar como causas del fracaso del proyecto si se hacen malas estimaciones.
	Son dificiles por que suelen realizarse al inicio del proyecto y es cuando menos informacion se tiene y el margen de error es mas grande.
	Para reducir la incertidumbre y mejorar la precision se debe seguir ciertas pautas 

	- Uso de datos historicos
		cuanta mas info de proyectos ya pasado se pueda recopilar va a aportar mas datos a esa estimacion
	
	- Evitar improvisacion
		una estimacion debe ser siempre fruto de un analisis y no se debe realizar sobre la marcha o bajo presion

	- Descomponer en tareas
		Es mejor estimar en tareas pequeñas que en un proyecto como un todo. La suma de esas tareas nos va a permitir hacer una estimacion global de todo.
		
		- Triangulacion de tecnicas
			No se debe confiar en un unico metodo de estimacion y es recomendable comparar resultados utilizando una comparacion con proyectos similares, utilizar modelos matematicos
		
		- Descomposicion funcional de tareas



	- Planificacion temporal y asignacion de recursos

		Planificar tareas por dias es contraproducente y obliga a  hacer reajustes de forma constante sobre cualquier imprevisto que surja. La solucion a esto es la planificacion de tareas semanales. 
Al tener una planificacion semanal permite absorber los improvistos y no colapsar el cronograma.
Se debe hacer un reajuste continuo que permita cambiar las planificaciones  por si por ejemplo, una tarea se retrasa. Lo que no se debe asumir es que se va a recuperar el tiempo usado. Eso puede generar una acumulacion de errores.


	- Mantener una comunicacion con el cliente para mantenerlo actualizado es una obligación

			- Garantizar la privacidad. evitar las interrupciones y disponer de un tiempo adecuado.

##### Puntos clave


- Se debe ser puntual, identificarse y explicar los objetivos de la entrevista

- Subrayar el tema de la entrevista y establecer la naturaleza del proyecto

- Comenzar con preguntas generales que establezcan el marco de trabajo que se usara en la entrevista.

- Continuar con los temas y aspectos que surgan de quienes responden. Asegurarse de encontrar por que quien responde cree que es tan importante el tema como para comentarlo.

- Limitar las notas que se escriban para evitar distraer a quien responde.

- Cuando todos los temas vistos con los entevistados se hayan discutido. relaizar otras preguntas especificas que se crea deban discutirse

- Se debe ser correcto y no preguntar cosas que se pueden obtener por otras vias a menos que se desee comprobar algo
	-  El tacto y la imparcialidad ayudan a asegurar una entrevista exitosa.

 - Debe prestar la maxima atencion durante el desarrollo, crear un clima


- Diferenciar entre los hechos y las opiniones

- Evitar que la etrevista sea larga o se convierta en inoportuna.

- Al final, resumir la informacion recabada. 

- al despedirse debe mostrar agradecimiento y dejar cordinado un rencuentro por dudas.

# Revision de registros

implican el examen y evaluacion sistematica de documentos y datos registrados. Esta info permite realizar y evaluar informacion que nos permita utilizarla como fuente de datos para la realizacion de un proyecto de software sobre todo cuando lo que se quiere realizarson nuevas versiones de un software ya existente

### Proceso a revisar para la revision de registros

este proceso puede variardel contexto en el que se vaya a aplicar, osea de donde este la info guardada.

	- Planificacion
		permite definir el alcance y el objetivo de la revision, Tambien identificar los registros que deben realizarse

	- Establecer los plasos para hacer esa tarea y los recursos necesarios


	- Recopilacion
		-Reunir los registros que sean relevantes
		- Organizar y clasificar los registros de manera sistematica

	-Analisis
		-se deberia examinar registros al detalle
		- Comparar datos con normas
		- Identficar errores o inconsistencias 


	-Evaluacion
		- Evaluar la fiabilidad y calidad de lso registros obtenidos
		- Determinar el impacto delos errores o irregularidades que se identificaron


	 -Comunicacion
		 - documentarlos allazgos realizados y tambien presentar los resultados a las partes interesadas

	- Acciones
		- se ejecutan las acciones que se definen el resultado de la evaluacion

## Anteproyecto

Es un documento inicial que describe a grandes rasgos un proyecto de software antes de que se inviertan recursos significativos en su desarrollo.
Este documento es flexible osea que sele pueden hacer modificaciones hasta llegar a una version definitiva.
Funciona como un plan preeliminar en el cual se establece la direccion del proyecto, los objetivos, el alcance, los requisitos basicos y la viabilidad 

### Componentes
	-Definicion inicial
			En la cual se establece una vision clara del software que se vaa desarrollar y tambien se definen los objetivos y el alcance del proyecto.

	- Viabilidad 
			Debe contener una evaluacion de la viabilidad tecnica y econoimica del proyecto y tambien identificar posibles riesgos que pueden aparecer.


	-Comunicacion
			Debe servir como un documento de referencia para todas las partes interesadas (Desarrolladores, clientes, inversores).
			Debe facilitar la comunicaciony el entendimiento común del proyecto


	-Planificacion preliminar
			-Cornograma y presupuesto inicialse deben establecer, ademas se deben definir las principales fases para el desarrollo


	-Requisitos 
		Una descripcion general de los requisitos funcionales y no funcionales del software y tambien deberia tener una identificaciondelos usuarios y sus necesidades


	-Arquitectura propuesta
			es la cual se establece el esquema general de la arquitectura de software y mencionar una seleccion de tecnologias y herramientas a utilizar


## conclusión

Un anteproyecto es una herramienta crucial para la planificacion inicial de un proyecto ya que ayuda a que todas las partes interesadas esten alineadas y que sea una base olida para la proxima etapa
